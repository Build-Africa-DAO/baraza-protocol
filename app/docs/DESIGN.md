# Baraza Protocol Design Direction

## Product Feel

Baraza Protocol should feel like a relaxed, trustworthy community-finance product with a polished blockchain layer underneath. The design should be warm, social, and grounded in African community savings culture, while still feeling modern enough for Solana, on-chain treasury records, and wallet-based governance.

The visual mood is calm confidence: a product for organisers, treasurers, SACCO members, chamas, welfare groups, and co-operatives who need shared visibility over money, votes, and membership.

## DAO Language Direction

Baraza should present DAO concepts in a human, community-first way. The product is not just a crypto dashboard. It is a DAO operating layer for real-world groups that need membership, treasury visibility, proposals, voting, quorum, and transparent execution.

Use DAO-related words naturally across pages:

- Community DAO

- DAO treasury
- On-chain governance

- Member voting

- Smart contract execution
- Governance rules

Preferred framing:

- "Create a Community DAO" instead of only "Start a Group"
- "DAO Treasury" instead of only "Community Fund"
- "Governance Proposals" instead of only "Decisions"
- "Member Voting" instead of only "Vote"
- "Membership Credential" instead of only "NFT"
- "On-chain audit trail" instead of only "history"

Avoid making the product feel abstract or over-technical. Pair DAO terms with familiar language:

- Chama DAO
- SACCO governance
- Welfare group treasury
- Co-operative voting
- Member-owned community fund
- M-Pesa to DAO treasury

## DAO Product Positioning

Baraza Protocol helps African communities create and operate phone-first DAOs. Each community can collect contributions, verify membership, manage a shared DAO treasury, create governance proposals, vote as members, and release funds only when community rules are satisfied.

The product should communicate that DAO governance can feel familiar: members contribute, discuss, vote, and keep records together. The blockchain layer provides transparent treasury records, verifiable membership, vote receipts, proposal accounts, and an immutable audit trail.

Short positioning line:

Create a Community DAO for your chama, SACCO, welfare group, or co-operative.

Expanded positioning:

Baraza turns real-world community funds into transparent Community DAOs with M-Pesa contributions, Solana membership credentials, governance proposals, member voting, and auditable DAO treasury records.

## 3D Background Concept

Use a soft 3D world globe in the background of the hero area. The globe should sit behind the main dashboard or headline content, large enough to create atmosphere but subtle enough that text and interface controls remain readable.

The globe should suggest global reach while keeping Africa visually meaningful. The African continent can be softly highlighted with warm gold or amber light. Around the globe, use gentle curved connection lines, small glowing nodes, and quiet orbit paths to represent community networks, M-Pesa payments, blockchain confirmations, member records, treasury activity, and voting signals.

The scene should feel relaxed, not intense. It should look like a calm evening dashboard for community organisers managing shared funds, not a speculative trading product.

## Composition

- Place the 3D globe slightly behind the hero/dashboard area.
- Prefer right-side or lower-background positioning so the headline has clean negative space.
- Keep the center-left area readable for hero text and CTA buttons.
- Add faint floating UI panels for votes, treasury balances, member records, and payment confirmations.
- Use soft depth of field so the globe supports the interface rather than competing with it.
- Keep all effects subtle on mobile so the content stays fast and clear.

## Layers and Connectors

The 3D hero background should be built as a set of quiet visual layers. Each layer should have a clear role, so the final scene feels rich without becoming noisy.

### Layer 1: Base Atmosphere

The base layer is a deep charcoal or near-black background with very subtle radial lighting. It should create depth behind the product UI without reading as a decorative gradient.

Use:

- Deep charcoal background
- Soft amber light falloff behind the globe
- Very faint green highlights near payment or treasury activity
- Low-opacity dust or particle texture only if it stays subtle

Avoid:

- Bright full-screen gradients
- Heavy noise
- Large decorative blobs
- Strong purple or blue dominance

### Layer 2: 3D Globe

The globe is the main background object. It should sit behind the foreground interface and be partially softened by depth of field.

Globe details:

- Africa should be visible and softly emphasized.
- Continents should be abstract but recognizable.
- The globe should use warm gold rim light and low-contrast surface detail.
- Oceans should stay dark, calm, and matte.
- Rotation should be slow and optional.

The globe should communicate global infrastructure and African-rooted coordination, not travel, surveillance, or trading.

### Layer 3: Network Nodes

Nodes represent members, communities, wallets, payment confirmations, and governance actions.

Node types:

- Member node: small warm gold dot
- Community node: larger amber dot with soft outer ring
- Treasury node: muted green dot with steady glow
- Vote node: small off-white or pale gold dot
- Payment node: gold dot that briefly pulses when active
- Chain confirmation node: tiny green confirmation sparkle

Nodes should be sparse. A few meaningful points are better than a crowded constellation.

### Layer 4: Connectors

Connectors are curved lines between nodes. They should feel like soft relationship paths, not hard technical diagrams.

Connector types:

- Community-to-member connector: thin amber arc
- Member-to-vote connector: pale gold arc
- Payment-to-treasury connector: muted green arc
- Treasury-to-proposal connector: gold-green arc
- Globe orbit connector: long, low-opacity line wrapping around the globe
- Cross-community connector: faint dotted arc showing network reach

Connector behavior:

- Lines should be low opacity by default.
- Active connectors can have a gentle moving highlight.
- Motion should be slow, relaxed, and non-distracting.
- Connectors should never cross directly over primary text.
- On mobile, reduce connector count and hide the longest arcs.

### Layer 5: Floating Product Panels

Floating panels should feel like transparent dashboard fragments from the actual product. They can sit around the globe, behind or beside the main hero content.

Panel examples:

- Treasury Balance: KSh 2,450,000
- Members Active: 248
- Proposal Vote: 72% Support
- Payment Confirmed: M-Pesa
- Membership Minted: Solana
- Audit Trail: Verified

Panel style:

- Dark translucent surface
- Thin amber or charcoal border
- Small icon or status dot
- Compact typography
- No large paragraphs
- No fake exchange charts

Panels should reinforce the product workflow: collect, govern, release, audit.

### Layer 6: Foreground Interface

The foreground is the actual product UI: headline, CTA buttons, dashboard preview, wallet state, community stats, and proposal cards.

The background must support this layer. If the globe, connectors, or floating panels compete with buttons, text, forms, or cards, reduce their opacity, blur, scale, or count.

Foreground priority:

1. Headline readability
2. CTA clarity
3. Dashboard preview
4. Trust and status indicators
5. Background atmosphere

### Connector Meaning Map

Use this meaning map when designing visual connectors:

| Visual Element | Product Meaning |
| --- | --- |
| Gold node | Member, community, or governance action |
| Green node | Confirmed payment, treasury state, or successful chain confirmation |
| Pale line | Membership or voting relationship |
| Green line | Payment or treasury movement |
| Dotted orbit | Blockchain audit trail or cross-community visibility |
| Slow pulse | New confirmation or active proposal |
| Outer ring | Community hub or shared treasury |
| Floating card | Product state from the Baraza dashboard |

### Layer Stack

Recommended visual stack from back to front:

1. Base charcoal atmosphere
2. Soft ambient glow
3. 3D globe
4. Globe surface lights
5. Network nodes
6. Curved connectors and orbit lines
7. Floating product panels
8. Dark readability overlay, if needed
9. Main hero text and CTA buttons
10. Dashboard preview or product cards

### Motion Rules

Motion should make the product feel alive, not busy.

- Globe rotation: very slow, 60 seconds or longer per rotation
- Connector highlight: slow travel, 6 to 12 seconds per path
- Node pulse: occasional and staggered
- Floating panels: tiny vertical drift only
- Particle movement: minimal or disabled
- Reduced motion: freeze globe, remove connector travel, keep static glow

### Mobile Simplification

On mobile:

- Hide floating panels that compete with headline text.
- Reduce connectors to 2 or 3 visible arcs.
- Keep only the most important nodes.
- Position the globe lower or further back.
- Increase text contrast with a subtle dark overlay.
- Disable heavy animation on low-power devices.

## Page Design System

The 3D globe and connector language should influence the whole product, but each page should use it with different intensity. The landing page can be atmospheric. Workflow pages should be calmer, denser, and more task-focused.

### Global Page Rules

- Use the same dark Baraza shell across all pages.
- Keep navigation simple, sticky where useful, and readable over dark backgrounds.
- Use warm gold for primary actions and important governance moments.
- Use muted green for confirmed payment, active membership, successful treasury state, and verified chain state.
- Use muted red only for destructive or failed states.
- Use globe and connector visuals as background atmosphere, not as content.
- Use dashboard cards, tables, tabs, and forms with compact spacing and clear hierarchy.
- Avoid making every page look like a landing page.

### Page Background Intensity

| Page | Background Intensity | Globe Use | Connector Use |
| --- | --- | --- | --- |

| Communities | Medium | Faint globe watermark | Light connector texture |
| Create Community | Low | Abstract glow only | Minimal or none |

| Join / Membership | Medium | Small globe behind status tracker | Payment-to-membership connector |

## End-to-End User Flows

### Flow 1: First-Time Visitor To Community DAO

1. User lands on `/`.
2. User sees the 3D globe, Community DAO positioning, DAO treasury preview, and CTA buttons.
3. User chooses `Browse Community DAOs`.
4. App routes to `/communities`.
5. User searches or filters by savings group, SACCO, cooperative, welfare group, housing, professional, or investment club.
6. User opens a community card.
7. App routes to `/dashboard/:id`.
8. User views DAO treasury, members, governance proposals, active votes, and membership fee.
9. User chooses `Join DAO` or `Join Group`.

Required screens:

- Landing
- Communities directory
- Community dashboard
- Join modal or `/join/:id`

### Flow 2: Phone-First M-Pesa Join

This is the most important missing MVP flow for non-crypto-native users.

1. User opens `/join/:id` from a community invite link.
2. User sees community name, DAO treasury summary, membership fee, and trust notes.
3. User enters phone number.
4. User verifies OTP.
5. User confirms membership fee.
6. App starts M-Pesa STK push.
7. User sees payment status screen.
8. Backend receives payment webhook.
9. Backend creates or confirms payment attestation.
10. App shows `Payment Confirmed`.
11. User links or creates wallet.
12. Smart contract mints membership credential and creates MemberAccount.
13. App shows `Chain Confirmed`.
14. User becomes `Active DAO Member`.
15. App routes to `/dashboard/:id`.

Status steps:

- Phone Entered
- OTP Verified
- Payment Started
- Payment Confirmed
- Wallet Linked
- Membership Minting
- Chain Confirmed
- Active DAO Member

Required screens:

- `/join/:id`
- `/join/:id/status`
- `/onboard`
- Community dashboard membership tab

### Flow 3: Wallet-Native Join

1. User opens `/dashboard/:id`.
2. User clicks `Join DAO`.
3. App asks user to connect wallet.
4. User connects Phantom, Solflare, or Coinbase Wallet.
5. Join modal shows membership fee in KSh and equivalent on-chain transaction context.
6. User confirms.
7. App submits join transaction.
8. App waits for chain confirmation.
9. App records active membership.
10. Membership card appears in `My Membership`.

Current repo support:

- Community dashboard exists.
- Wallet guard exists.
- Join modal exists.
- Prototype local membership recording exists.

Needed additions:

- Real transaction status stepper
- Payment attestation state
- MemberAccount confirmation
- Failure and retry states

### Flow 4: Create A Community DAO

1. User clicks `Create a Community DAO`.
2. App routes to `/create`.
3. User connects wallet if not already connected.
4. User enters community name.
5. User selects community type.
6. User enters monthly contribution in KSh.
7. User enters community description.
8. User submits form.
9. App creates community record.
10. App shows success state.
11. User opens `/dashboard/:id`.
12. Future implementation creates DAO treasury account, governance rules, and membership settings.

Current repo support:

- `/create` exists.
- Wallet gating exists.
- Form exists.
- Success screen exists.

Needed additions:

- DAO rule fields: quorum, approval threshold, voting period default
- Treasury account setup status
- Membership credential settings
- On-chain setup tracker

### Flow 5: Create Governance Proposal

1. Active member opens `/dashboard/:id`.
2. User selects `Governance Proposals`.
3. User clicks `New Decision` or `Create Proposal`.
4. App routes to `/dashboard/:id/decisions/create`.
5. User enters proposal title.
6. User enters proposal description.
7. User enters requested funding amount.
8. App validates amount against DAO treasury balance.
9. User selects voting period.
10. User submits proposal.
11. App creates ProposalAccount on-chain.
12. Proposal appears in active votes.

Current repo support:

- Create decision page exists.
- Budget validation exists.
- Wallet guard exists.
- Not-wired toast exists.

Needed additions:

- Active member guard
- Proposal persistence
- ProposalAccount creation
- Quorum and approval threshold display
- Proposal detail page

### Flow 6: Vote On A Proposal

1. Active DAO member opens `/dashboard/:id`.
2. User opens active governance proposal.
3. User reviews funding amount, treasury impact, quorum, deadline, and vote history.
4. User clicks `Support` or `Object`.
5. Wallet prompts signature.
6. App submits vote transaction.
7. Smart contract creates VoteReceiptAccount.
8. UI updates optimistically.
9. App confirms indexed chain state.
10. Proposal tally updates.

Needed screens/components:

- Proposal detail page
- Vote confirmation modal
- Vote receipt state
- Quorum progress component
- Wallet signature status

### Flow 7: Treasury Release After Vote

1. Proposal voting period ends.
2. App checks quorum and approval threshold.
3. If approved, proposal enters `Ready for Treasury Release`.
4. Authorized signer or smart contract executor triggers treasury release.
5. App shows transaction pending.
6. Chain confirms transfer.
7. Treasury balance updates.
8. DAO audit trail records release.

Needed screens/components:

- Treasury detail page
- Proposal execution state
- Release confirmation modal
- DAO audit trail table
- Transaction explorer link

### Flow 8: Admin Reconciliation

1. Admin opens `/admin`.
2. Admin views payment orders, mint jobs, failed webhooks, refunds, and manual review items.
3. Admin filters by status.
4. Admin opens a stuck order.
5. Admin retries mint, marks refund required, or resolves manually.
6. App records admin action and audit note.

Needed screens/components:

- Admin dashboard
- Payment order detail drawer
- Mint job status table
- Refund queue
- Manual review queue
- Admin action audit log

## Wallet, Payment, Backend, And Frontend Architecture

This repo currently contains a Vite React frontend in `app/` with Solana wallet scaffolding, mock DAO data, local/Supabase community records, and guarded transaction hooks. A production MVP still needs a backend service for phone identity, M-Pesa payment orders, webhooks, payment attestation, mint jobs, refunds, and admin reconciliation.

### Current Frontend Reality

Already present in the repo:

- Public routes: `/`, `/communities`, `*`
- Wallet-gated routes: `/create`, `/dashboard/:id`, `/dashboard/:id/decisions/create`
- Wallet provider support through `WalletProviders`
- Wallet state UI through `WalletStatus`
- Public lazy wallet slot through `PublicWalletSlot`
- Wallet gating through `useWalletGuard`
- Solana write scaffolding through `useBarazaContract`
- RPC fallback through `withRpcFallback`
- Community data loading through `useCommunities`
- Supabase community table support with local fallback
- Prototype local membership record support

Important current limitation:

- There is no backend payment service in the repo yet.
- There is no Africa's Talking webhook handler yet.
- There is no real payment order table beyond architecture docs.
- There is no mint worker yet.
- There is no durable membership activation flow yet.
- `joinCommunity`, proposal creation, and voting need real program instructions and durable persistence before they can be treated as final product actions.

### Frontend Responsibilities

The frontend should own user experience, wallet interaction, visible status, form validation, and user-safe state transitions.

Frontend must handle:

- Wallet connect, disconnect, copy address, and change wallet
- Wallet-required action gating
- Wrong-network warning
- DAO browsing and filtering
- Community DAO creation form
- Join DAO modal and phone-first join screens
- M-Pesa payment request UI
- Payment status tracker
- Wallet onboarding or link prompts
- Membership card display
- Governance proposal creation form
- Proposal detail and vote UI
- Treasury dashboard and audit trail views
- Admin dashboard UI, once backend endpoints exist
- Loading, empty, timeout, retry, failed, and success states

Frontend must not:

- Store payment provider secrets
- Trust client-side payment success as final
- Mark membership active after M-Pesa confirmation alone
- Mint membership directly from a public client with privileged authority
- Expose phone numbers, M-Pesa receipts, provider payloads, admin notes, or private KYC data

### Backend Responsibilities

The backend is the source of truth for off-chain payment, reconciliation, identity, mint jobs, admin review, and private records.

Backend must handle:

- Phone OTP verification
- User identity records
- Wallet link records
- Payment order creation
- Africa's Talking M-Pesa request creation
- Provider webhook verification
- Payment event storage
- Duplicate webhook detection
- Amount and currency validation
- Payment order state transitions
- Mint job queueing
- Payment attestation creation
- Membership mint submission
- Indexer or devnet confirmation polling
- Membership activation
- Refund queue
- Admin manual review
- Audit events

Backend must keep these server-side only:

- Africa's Talking username and API key
- Webhook signing or validation secrets
- Payment signer key
- Mint worker signer key or program authority controls
- Phone encryption key
- Phone HMAC pepper
- Admin tokens
- Provider raw payloads

### Recommended Backend Modules

Use a small API service before adding broad infrastructure.

Suggested modules:

| Module | Responsibility |
| --- | --- |
| `identity` | Phone OTP, user records, wallet links |
| `payments` | Payment order creation and provider request |
| `webhooks` | Africa's Talking webhook intake and validation |
| `reconciliation` | Payment event normalization, duplicate handling, amount checks |
| `mint-worker` | Payment attestation, membership mint, retry logic |
| `indexer` | Helius webhook or devnet polling confirmation |
| `membership` | Member activation, status, renewal, suspension |
| `governance` | Proposal and vote persistence/read models |
| `admin` | Refunds, manual review, retry actions, audit trail |

### Frontend To Backend Join Contract

The phone-first join flow should use explicit states so users always understand what is happening.

Frontend request:

```text
POST /api/communities/:id/join-orders
```

Request body:

```json
{
  "phone": "+254700000000",
  "walletAddress": "optionalSolanaWallet",
  "membershipTierId": "basic",
  "returnUrl": "https://baraza.finance/join/community-id/status"
}
```

Backend response:

```json
{
  "orderId": "ord_123",
  "communityId": "community-id",
  "amount": 500,
  "currency": "KES",
  "status": "PAYMENT_REQUESTED",
  "provider": "africas_talking",
  "expiresAt": "2026-05-13T12:30:00.000Z"
}
```

Frontend then polls or subscribes:

```text
GET /api/payment-orders/:orderId
```

Status response:

```json
{
  "orderId": "ord_123",
  "status": "MINT_SUBMITTED",
  "paymentStatus": "PAYMENT_CONFIRMED",
  "mintStatus": "PENDING",
  "membershipStatus": "PENDING",
  "walletAddress": "userWallet",
  "transactionSignature": "optionalSignature",
  "nextAction": "WAIT_FOR_CHAIN_CONFIRMATION"
}
```

### Wallet Connection Flow

Wallet connection should be consistent across protected screens.

1. User clicks a wallet-required action.
2. `useWalletGuard` checks `connected` and `publicKey`.
3. If missing, open wallet modal.
4. User selects Phantom, Solflare, Coinbase Wallet, or supported adapter.
5. App stores wallet adapter state through wallet adapter auto-connect.
6. `WalletStatus` shows truncated address.
7. App checks network.
8. If network is wrong, show visible warning and block chain actions.
9. User can copy address, change wallet, or disconnect.

Required wallet UI states:

- Not connected
- Connecting
- Connected
- Wrong network
- Wallet unavailable
- Signature requested
- Signature rejected
- Transaction submitted
- Transaction confirmed
- Transaction failed

### Wallet-Native Payment Flow

Wallet-native payment should remain optional for MVP. It should not replace the M-Pesa path.

1. User connects wallet.
2. User selects membership tier.
3. App shows fee in KSh and crypto equivalent if supported.
4. User confirms wallet-native payment.
5. App creates payment order with `provider = solana_pay` or equivalent.
6. User signs transaction.
7. Backend or indexer verifies payment transaction.
8. Backend creates payment attestation or marks payment order confirmed.
9. Mint job creates membership credential.
10. Membership activates after mint and indexer confirmation.

Do not activate membership immediately after wallet transfer. The same reconciliation and mint confirmation rules apply.

### M-Pesa Payment Flow

M-Pesa payment is the primary MVP flow.

1. Frontend collects phone number and selected membership tier.
2. Backend creates `payment_order`.
3. Backend calls Africa's Talking M-Pesa checkout/STK push.
4. Frontend shows `Check your phone for the M-Pesa prompt`.
5. Africa's Talking sends webhook.
6. Backend verifies and stores raw payload hash.
7. Backend validates amount, currency, environment, provider reference, and order ID.
8. Backend transitions order to `PAYMENT_CONFIRMED`.
9. Backend queues mint job.
10. Frontend shows `Payment received, activating membership`.
11. Mint worker creates payment attestation.
12. Mint worker mints membership credential.
13. Indexer confirms mint.
14. Backend activates membership.
15. Frontend shows membership card and routes to dashboard.

Hard rule:

Payment confirmed does not mean membership active. The UI must keep these as separate states.

### Payment Attestation Flow

Payment attestation bridges off-chain payment to on-chain membership minting.

1. Backend confirms payment order.
2. Backend signs or submits a PaymentAttestationAccount on Solana.
3. Attestation includes order hash, community, tier, amount, currency, recipient wallet, provider environment, expiry, consumed state, and attestation signer.
4. Membership mint instruction consumes the attestation.
5. Consuming the attestation and minting membership must be atomic.
6. Attestation can be consumed only once.
7. Duplicate webhook cannot create a second active attestation or second membership.

Frontend should display only safe summaries:

- Payment received
- Membership activating
- Chain transaction pending
- Membership verified

Frontend should not display raw provider payloads or sensitive payment references.

### Membership Activation Flow

Membership becomes active only after all of these are true:

```text
payment_order.status >= PAYMENT_CONFIRMED
payment_attestation exists
membership mint transaction submitted
mint confirmed by RPC
Helius/indexer or read model confirms member account
backend membership.status == ACTIVE
```

Access check:

```text
wallet connected
AND wallet address matches active MemberAccount
AND membership.status == ACTIVE
AND community_id matches
AND proposal snapshot allows action
```

### Security Rules

- No `VITE_` variable may contain backend secrets.
- Frontend may use Supabase anon key only for public/RLS-safe operations.
- Backend must use service keys only server-side.
- Phone numbers must be encrypted or HMAC/peppered server-side.

- Payment webhooks must be durable before returning success to provider.
- Duplicate webhooks must be safe no-ops.
- One payment order can mint at most one active membership.
